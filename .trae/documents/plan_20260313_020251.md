# 优化 GitLab Merge Request Diff 获取逻辑 (Retry + Compare API)

**问题确认**:
您提到的**时序问题非常准确**。GitLab 的 Merge Request 创建后，后台需要时间异步计算 Diff。如果 Webhook 触发过快，`/diffs` 接口可能返回空数据或不完整的 Diff，导致 `files_changed` 等统计为 0。这也解释了为什么 Push 事件（使用实时计算的 `/compare` 接口）是正常的。

**解决方案**:
为了彻底解决这个问题，我们将采取双重保障措施：

1. **切换到实时 Compare 接口**:
   将 MR 的 Diff 获取方式从依赖异步计算的 `/diffs` 接口，切换为与 Push 事件一致的 `/compare` 接口。该接口直接对比源分支和目标分支，**无需等待 GitLab 后台任务完成**，能立即获取准确的 Diff。

2. **增加重试机制 (Retry)**:
   正如您建议的，在获取 Diff 时增加重试逻辑。如果首次获取失败或为空，系统将等待短暂时间后重试，确保在网络波动或极端的时序情况下也能获取到数据。

**实施计划**:
1. 修改 `backend/internal/services/webhook/gitlab.go`。
2. 引入 `time` 包。
3. 重构 `processGitLabMR` 函数：
   - 优先使用 `getGitLabCompareDiff` 获取 Diff。
   - 添加重试循环（最多重试 3 次，每次间隔 2 秒）。
   - 如果 `Compare` 接口依然失败，作为兜底方案回退到 `getGitLabMRDiff`（同样带重试）。
